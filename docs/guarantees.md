# Formal Guarantees

[Docs home](README.md) · [Project README](../README.md)

This page is the central map of the theorem-backed guarantees in Lean UFO. It
separates three things that are easy to conflate:

- what the Lean kernel checks as proof evidence;
- what the executable DSL compiler and checker compute;
- what remains trusted frontend or diagnostic presentation code.

The main theorem files are:

```text
LeanUfo/UFO/Core/*.lean
LeanUfo/UFO/Models/*.lean
LeanUfo/UFO/DSL/Guarantees.lean
LeanUfo/UFO/DSL/Checker/Soundness.lean
LeanUfo/UFO/DSL/Checker/Complexity.lean
LeanUfo/UFO/DSL/Certification.lean
```

## Core Semantic Layer

The core formalization defines UFO signatures and axiom packages as ordinary
Lean propositions over S5 Kripke semantics. These files are not DSL-specific:
they are the semantic target that DSL models must satisfy.

Representative artifacts:

```lean
UFOSignature3_1
UFOAxioms3_1
...
UFOSignature4
UFOAxioms4
```

The formal guarantee is direct: any theorem of type `UFOAxioms4 Sig` is a Lean
proof that the Prop-valued semantic signature `Sig` satisfies the encoded UFO
axiom package through Section 4.

The core also contains derived semantic theorems, especially in
`Core/S5_Derived.lean`. These are not generated by the DSL. They document what
follows from the chosen S5 semantics and from explicit bridge axioms such as
kind stability.

Guarantee offered:

- core theorem statements are checked by Lean's kernel;
- modal and taxonomic results are ordinary propositions over the semantic
  signatures;
- no DSL parser or checker is involved in these proofs.

## Witness Models and Consistency Checkpoints

The `Models/` layer constructs concrete semantic witnesses for implemented UFO
fragments. These establish joint satisfiability checkpoints, for example:

```lean
consistent_3_1 :
  ∃ (Sig : UFOSignature3_1.{0,0}), UFOAxioms3_1 Sig

consistent_4 :
  ∃ (Sig : UFOSignature4.{0,0}), UFOAxioms4 Sig
```

These are semantic model-existence theorems. They are not proof-theoretic
consistency theorems for a UFO proof calculus; no such proof calculus is
implemented. The guarantee is that Lean has checked a concrete model satisfying
the packaged axioms.

Guarantee offered:

- the implemented axiom packages have concrete satisfying interpretations;
- each checkpoint is relative to Lean's metatheory and the selected semantic
  encoding;
- witness models are independent of the finite DSL command frontend.

## DSL Frontend Boundary

The DSL begins with surface syntax:

```lean
ufo_model ModelName : UFO where
  worlds actual
  things Person Alice
  given actual:
    ObjectKind(Person)
    Object(Alice)
    Alice :: Person
  derive_relations
  certify
```

The parser and command elaborator live mainly in:

```text
LeanUfo/UFO/DSL/Frontend/SurfaceSyntax.lean
LeanUfo/UFO/DSL/Syntax.lean
```

This layer is trusted metaprogramming. Lean checks the declarations emitted by
the command, but we do not currently prove a full correctness theorem for the
parser itself.

The frontend emits reusable declarations such as:

```lean
Model.source              : ModelSource
Model.data                : FiniteModel4
Model.sig                 : UFOSignature4
Model.checked_axN         : Checker.checkAxN Model.data = true
Model.certified_axN       : ax_aN ...
Model.certified           : UFOAxioms4 Model.sig
Model.certifiedModel      : FiniteModel4.Certified Model.data
Model.certificateManifest : CertificateManifest
```

Guarantee offered:

- the parser/emitter is trusted;
- the emitted declarations are checked by Lean;
- proof evidence is the generated Lean theorem, not the diagnostics widget or
  certificate manifest.

## Name Resolution and Scope Expansion

After parsing, name and scope handling is ordinary Lean code in the compiler.
The relevant guarantee namespace is:

```lean
LeanUfo.UFO.DSL.NameResolution
LeanUfo.UFO.DSL.ScopedCompiledFact
```

Representative theorems:

```lean
resolveThing_of_nameIndex_eq_some :
  nameIndex? ts thing = some idx ->
  resolveThing ts thing = Except.ok idx

resolveThing_of_nameIndex_eq_none :
  nameIndex? ts thing = none ->
  resolveThing ts thing = Except.error (.unknownThing thing)

resolveWorld_of_nameIndex_eq_some :
  nameIndex? ws world = some idx ->
  resolveWorld ws world = Except.ok idx

resolveWorld_of_nameIndex_eq_none :
  nameIndex? ws world = none ->
  resolveWorld ws world = Except.error (.unknownWorld world)

resolveScope_everywhere :
  resolveScope ws .everywhere = Except.ok .everywhere

unary_at_expands_to_singleton
binary_at_expands_to_singleton
ternary_at_expands_to_singleton
tupleProjection_at_expands_to_singleton
scoped_expansion_pipeline
```

These theorems state that successful name lookups resolve to the returned
finite index, failed lookups produce explicit compiler errors, and world-scoped
facts expand through the pure compiler pass rather than ad-hoc frontend logic.

Guarantee offered:

- unknown names are not silently accepted;
- `everywhere` is preserved as an explicit scope before expansion;
- scoped facts have theorem-backed expansion behavior.

## Compiler and Table Construction

The pure compiler turns resolved facts into finite tables:

```text
NamedScopedFact
  -> ScopedCompiledFact
  -> CompiledFact
  -> ModelAST
  -> FactTables
  -> FiniteModel4
```

Representative theorem namespaces:

```lean
LeanUfo.UFO.DSL.CompiledFact
LeanUfo.UFO.DSL.ModelAST
LeanUfo.UFO.DSL.FactTables
```

Representative theorems:

```lean
unary_compiles_with_taxonomy
binary_compiles_to_table
ternary_compiles_to_table
tupleProjection_compiles_to_table
derived_compiles_to_assertion

explicit_unary_compiles_to_table
explicit_binary_compiles_to_table
explicit_ternary_compiles_to_table
explicit_tupleProjection_compiles_to_table
explicit_derived_compiles_to_assertion

taxonomy_expansion_pipeline
reflexive_specialization_expansion_pipeline
compilation_pipeline
explicit_compilation_pipeline
explicit_model_pipeline
```

These theorems record how explicit facts enter the compiled tables and how the
main compiler pipeline is assembled. They are intentionally local engineering
guarantees, not a full verified compiler theorem from concrete syntax to
semantics.

Guarantee offered:

- explicit unary, binary, ternary, and tuple-projection facts become the
  intended table entries;
- derived assertions are separated from primitive table facts;
- taxonomy and reflexive-specialization passes are explicit compiler stages;
- finite model construction is performed by ordinary Lean functions.

## Finite Model Semantic Bridge

`FiniteModel4` is the executable finite representation. The bridge into the
ordinary semantic axiom package is:

```lean
FiniteModel4.toUFOSignature4 : UFOSignature4
```

The main guarantee namespace is:

```lean
LeanUfo.UFO.DSL.FiniteModel4
```

Representative theorems:

```lean
certified_iff_ufoAxioms4 :
  M.Certified ↔ UFOAxioms4 M.toUFOSignature4

certified_sound :
  M.Certified -> UFOAxioms4 M.toUFOSignature4

certified_of_ufoAxioms4 :
  UFOAxioms4 M.toUFOSignature4 -> M.Certified

compiled_frame_universal :
  M.toS5Frame.R w v

compiled_signature_frame_universal :
  M.toUFOSignature4.F.R w v

compiled_inst_iff :
  M.toUFOSignature4.Inst x y w ↔ M.inst x y w = true

compiled_sub_iff :
  M.toUFOSignature4.Sub x y w ↔ M.sub x y w = true

compiled_memberOf_iff_setExtension :
  MemberOf M.toUFOSignature4.toUFOSignature3_12 x s w ↔
    M.memberOf x s w = true

compiled_constitution_iff :
  M.toUFOSignature4.Constitution x x' y y' w ↔ ...
```

These theorems explain how table values are interpreted as semantic predicates.
For instance, `compiled_inst_iff` says that semantic instantiation in the
compiled signature is exactly the Boolean instantiation table. The frame
theorems also make explicit that generated DSL models currently use a universal
S5 accessibility relation.

Guarantee offered:

- finite tables are connected to the Prop-valued UFO signature;
- `FiniteModel4.Certified` is not a new semantics, but exactly `UFOAxioms4`
  for the compiled signature;
- current DSL frames are universal S5 frames.

## Reflective Checker Soundness

The checker is an executable Boolean validator over `FiniteModel4`:

```lean
checkAxN     : FiniteModel4 -> Bool
checkAxioms4 : FiniteModel4 -> Bool
```

The main theorem file is:

```text
LeanUfo/UFO/DSL/Checker/Soundness.lean
```

Representative per-field theorem pattern:

```lean
checkAxN_sound :
  checkAxN M = true ->
  ax_aN M.toUFOSignature4...
```

The aggregate theorem is:

```lean
checkAxioms4_sound :
  checkAxioms4 M = true ->
  UFOAxioms4 M.toUFOSignature4
```

This is the central certification theorem for successful DSL models. Generated
certificate fields have the shape:

```lean
theorem Model.certified_axN : ax_aN Model.sig... :=
  Checker.checkAxN_sound Model.data (by native_decide)
```

and the final generated theorem packages the per-field proofs:

```lean
Model.certified : UFOAxioms4 Model.sig
```

Guarantee offered:

- if the checker returns `true`, the corresponding semantic axiom holds;
- if the aggregate checker returns `true`, the compiled finite model satisfies
  `UFOAxioms4`;
- successful certification no longer depends on open-ended Lean tactic proof
  search over a large unfolded generated proposition.

## Completeness, Negative Witnesses, and ax99

For many fields, the checker also proves the converse:

```lean
checkAxN_complete :
  ax_aN M.toUFOSignature4... ->
  checkAxN M = true

checkAxN_iff :
  checkAxN M = true ↔ ax_aN M.toUFOSignature4...
```

Completeness is what lets a failed Boolean check become a confirmed negative
witness. If `checkAxN M = false` and `checkAxN_complete` is available, Lean can
prove `¬ ax_aN ...` by contradiction.

Hard cases have specialized guarantees:

```lean
checkAx68_sound
checkAx68_complete
checkAx68_correct
```

For `ax68`, the checker uses a bounded finite closure over `InheresIn`, and the
closure is proved sound and complete with respect to the core inductive
`MomentOf` interpretation.

`ax99` is different. The core axiom quantifies over arbitrary product-family
witnesses, while the finite checker can only inspect stored finite witnesses.
The checker therefore exposes:

```lean
ax99Finite

checkAx99_correct_finite :
  checkAx99 M = true ↔ ax99Finite M

ProductFamilyWitnessTableComplete M

not_ax99_of_checkAx99_false :
  ProductFamilyWitnessTableComplete M ->
  checkAx99 M = false ->
  ¬ ax_a99 M.toUFOSignature4.toUFOSignature3_12
```

The exact theorem names may be read in `Checker/Soundness.lean`; the important
distinction is stable: `checkAx99 = false` means "no stored finite witness was
found" unless the explicit representation-completeness condition is available.

Guarantee offered:

- direct-complete fields support confirmed semantic negative witnesses;
- `ax68` closure is not an approximation of `MomentOf`;
- `ax99` is sound for positive certification, but negative semantic
  interpretation requires explicit finite-witness completeness.

## Certificate Reuse and Model Extension

Certified models emit reusable atoms:

```lean
Model.checked_axN : Checker.checkAxN Model.data = true
```

An extended model may reuse a parent check only by producing a Lean theorem that
transports the parent Boolean result to the child:

```lean
checkAxN Child.data = checkAxN Parent.data
```

The planner for reuse lives in:

```text
LeanUfo/UFO/DSL/Certificate/Reuse.lean
```

The formal proof pattern is recorded in `Guarantees.lean`:

```lean
reused_checker_result_sound :
  check Child = check Parent ->
  check Parent = true ->
  check Child = true

reused_checker_semantic_sound :
  (∀ M, check M = true -> propOf M) ->
  check Child = check Parent ->
  check Parent = true ->
  propOf Child

reused_aggregate_checker_certified_sound :
  Checker.checkAxioms4 Child = Checker.checkAxioms4 Parent ->
  Checker.checkAxioms4 Parent = true ->
  UFOAxioms4 Child.toUFOSignature4

certificateReuseSource_fresh_none :
  certificateReuseSource? parentName parentSource childSource
    parentTables childTables true field = none
```

The first three theorems say that reuse does not introduce a new trusted
semantic route: it only supplies the same `check = true` premise that fresh
certification would supply. The last theorem says that `certify_fresh` disables
reuse planning.

The certificate manifest records whether a field was fresh or reused. The
manifest is audit metadata, not proof evidence. It can be exported with:

```bash
lake exe export-certificates --module Module.Name --out certificates/
lake exe validate-certificate certificates/Model.certificate.json --structure-only
lake exe validate-certificate certificates/Model.certificate.json --module Module.Name
```

`export_certificate Model` marks selected models for export. The exporter also
adds local git commit/tag metadata and SHA-256 source/model digests when
available. `--structure-only` checks manifest shape and required metadata
fields. The default validation path requires `--module`; it rebuilds the named
module, checks that the manifest's final theorem declarations have the expected
certificate types, and compares regenerated SHA-256 digests and theorem names.

Guarantee offered:

- reused checks are sound only through a Lean-checked equality proof;
- semantic correctness still comes from checker soundness;
- `certify_fresh` contributes no reuse source at the planner level;
- model-extension reuse cannot bypass the Lean kernel.

## Diagnostics Status Guarantees

Diagnostics explain success and failure states to the user. They are not proof
evidence, but the status classifier itself has small pure guarantees.

The main namespace is:

```lean
LeanUfo.UFO.DSL.Diagnostics
```

Representative theorems:

```lean
diagnostic_success_iff_completed :
  diagnosticCertStatus completed failed? field = "success" ↔
    field ∈ completed

diagnostic_failed_iff_recorded_failure_of_not_completed :
  field ∉ completed ->
  diagnosticCertStatus completed failed? field = "failed" ↔
    failed? = some field

diagnostic_unchecked_iff_not_recorded_failure_of_not_completed :
  field ∉ completed ->
  diagnosticCertStatus completed failed? field = "unchecked" ↔
    failed? ≠ some field

diagnostic_status_exhaustive :
  status = "success" ∨ status = "failed" ∨ status = "unchecked"
```

Guarantee offered:

- completed fields are displayed as successful;
- the first recorded failure is displayed as failed;
- later non-completed fields are displayed as unchecked;
- the widget does not invent additional certificate states.

Confirmed counterexamples rely on separate Lean-checked negation probes, not on
the widget.

## Decidability and Complexity

The reflective checker is executable Lean code returning `Bool`, so the checker
result is decidable by computation:

```lean
checkAxioms4 M : Bool
```

Through soundness, `true` is enough for a semantic certificate:

```lean
checkAxioms4_sound :
  checkAxioms4 M = true ->
  UFOAxioms4 M.toUFOSignature4
```

Through completeness theorems, a `false` result can also decide semantic
failure for the relevant field. For the full aggregate semantic package, the
current blocker to a completely unconditional equivalence is `ax99`: the finite
checker needs stored product-family witnesses, while the core axiom quantifies
over arbitrary witnesses. With `ProductFamilyWitnessTableComplete`, `ax99`
failure can be interpreted semantically; without it, failure means missing
finite witness data.

The abstract step model is:

```lean
structure Stepped (α : Type) where
  value : α
  steps : Nat
```

Value coherence:

```lean
checkAxioms4_S_value :
  (checkAxioms4_S M).value = checkAxioms4 M
```

Two-dimensional polynomial bound:

```lean
finite_checker_polynomial_bound :
  ∃ C a b,
    ∀ M : FiniteModel4,
      (checkAxioms4_S M).steps ≤
        C * (M.thingCount + 1)^a * (M.worldCount + 1)^b + 115
```

The current concrete witnesses are `C = 116`, `a = 4`, and `b = 2`.

One-variable model-size bound:

```lean
modelSize M = (M.thingCount + 1) * (M.worldCount + 1)

checkAxioms4_steps_polynomial_in_modelSize :
  ∃ C d k,
    ∀ M : FiniteModel4,
      (checkAxioms4_S M).steps ≤ C * (modelSize M)^d + k
```

The current concrete witnesses are `C = 116`, `d = 6`, and `k = 115`.

What this means:

- the semantic finite checker is no longer open-ended tactic search;
- the checked step model is polynomial in the generated finite model size;
- primitive table lookup is counted as constant in the abstract model;
- this is not a wall-clock bound for Lean, Lake, native compilation, kernel
  checking, editor diagnostics, or operating-system runtime.

## Trusted Boundary

The current trusted boundary is:

- Lean's kernel and the imported Lean/mathlib stack;
- the concrete DSL parser and declaration emitter in `Syntax.lean`;
- the fact that generated source strings correspond to the intended emitted
  declarations;
- the editor widget as presentation code.

The theorem-backed boundary is:

- name resolution and scope expansion as pure compiler functions;
- fact-to-table compilation properties;
- finite-table-to-signature bridge properties;
- checker soundness and selected completeness/correctness theorems;
- certificate reuse proof pattern;
- diagnostic status classification;
- abstract checker step bounds.

Generated proof declarations are always checked by Lean. The manifest and JSON
export layer are provenance and audit metadata; they are not proof evidence.
